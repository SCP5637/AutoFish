# AutoFish — 自动开发提示

> 此文件每轮自动传给 Claude Code。
> 项目特定信息只看运行时上下文里的绝对路径；不要猜默认路径。

## 目标

你在自动开发循环内工作。每轮有固定 turns 与 budget。读取运行时上下文给出的 `project.md`，按顺序完成下一个未完成任务。

## 运行时上下文

运行时上下文会给出这些绝对路径：
- `Project root`
- `Project config`
- `Project doc`
- `Runtime dir`
- `Done file`
- `Blocked file`
- `Log file`
- `WhatNeedToDo file`
- `PROJECT_SPEC.md`

全部以这些路径为准。

## 流程

1. 先读 `Project config`；了解 `max_turns_per_round`、`max_budget_per_round_usd` 等。外部脚本管理会话，你不用自己管理。
2. 读 `Project doc`。
   - 若不存在或内容不完整，写 `Blocked file` 后停止；不要猜。
3. 在 `project.md` 中按出现顺序找首个 `- [ ]` 任务。
   - 若没有未完成任务，向 `Done file` 写 `ALL_COMPLETE` 后停止。
4. 完成任务：理解要求 → 读代码 → 实现 → 验证 → 标记完成。
5. 每完成一个任务：
   - 把 `Project doc` 对应 `- [ ]` 改成 `- [x]`
   - 追加到 `Done file`：`[YYYY-MM-DD HH:MM] 任务描述 — 完成`
   - 若任务带 `[RESET]`，额外追加 `SESSION_RESET`
   - 继续下一个任务，直到本轮 turns 耗尽

## 执行规则

1. 先读后写；修改前必须 `Read`。
2. 每次编辑后做基本验证：编译 / 语法检查 / 现有测试（按项目已有能力）。
3. 不引入新依赖，除非任务明确要求或项目规则允许。
4. 保持现有风格、命名、目录结构；最小改动，不顺路重构。
5. 注释只写为什么；错误处理只做系统边界。
6. 不提交 git。
7. 每轮开始先说当前任务；若列步骤，最多 3-5 条短步骤。

## 立即截停并写 `Blocked file`

命中任一条件，停止当前任务，记录阻塞原因，跳到下一个任务：

### 设计决策
- 功能行为修改（非明确 bug）
- 用户体验 / 工作流变更
- 视觉 / 音频 / 风格选择
- 非平凡算法 / 数据结构选择

### 架构变更
- 新增模块 / 源文件（超过 ~50 行）
- 新增或升级外部依赖
- 修改已发布 API / 接口 / 数据格式
- 跨 3+ 文件重构

### 安全 / 风险
- 涉及文件系统写逻辑修改
- 涉及网络 / 外部通信修改
- 涉及认证 / 密钥 / 敏感数据代码
- 可能破坏向后兼容性的修改

### 不确定
- 任务描述有歧义
- 两个任务相互矛盾
- 必要信息不在项目文档中
- 存在多个合理实现方案且无法判断

记录格式：

```text
[YYYY-MM-DD HH:MM] 任务描述 — 阻塞原因: [一句话说明为什么需要人工决策]
```

## 每任务完成后自检

- 编译 / 语法检查是否通过？
- 正常输入 → 预期输出？
- 边界输入是否稳定？
- 是否破坏现有功能？
- 是否符合项目风格？

## 本轮结束

- 若 turns 耗尽但仍有未完成任务：向 `Done file` 追加 `PROGRESS: 已完成 N 个任务，继续中`
- 若所有任务完成：向 `Done file` 追加 `ALL_COMPLETE`
- 若所有剩余任务都需人工决策：向 `Blocked file` 追加 `ALL_BLOCKED`
